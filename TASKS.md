## Decisions log

- 2026-05-23 — **v10 mid-run assessment @ 2.8M steps → plan v11 with frame_stack=4 + LSTM hybrid.** v10 at 2.8M: `eval/mean_return=1993`, `eval/mean_length=555`, `clip_fraction=0.363`, `eval/std_return=1141`. For comparison, v9 at 3M: `eval/mean_return=7028`, `eval/mean_length=4500`. Three failure modes identified. **(1) frame_stack=1 cold-start blindness:** with `h=0,c=0` at episode reset, the LSTM has zero temporal context for the first several decision steps — the agent flies blind on every new episode. A single 84×84 grayscale frame contains no velocity information; the LSTM must infer motion direction from sequential observations, but this takes multiple steps to accumulate. v9 had immediate velocity context via frame stacking. This is the primary driver of `mean_length=555` (dying fast). **(2) Clip_fraction=0.363 oscillation:** stuck elevated since 1.2M steps (0.35 then, 0.36 now — no improvement). RecurrentPPO's LSTM re-execution drift accumulates across the 4 optimization epochs each rollout, causing the importance ratio `r_t = π_θ/π_θ_old` to drift outside [0.9,1.1] on a large fraction of transitions. The policy commits to a strategy, overshoots, gets clipped, oscillates. This is a structural problem with `n_epochs=4` for recurrent policies. **(3) Stochastic/deterministic gap:** rollout return 3311 vs eval return 1993 (1318-point gap). The argmax deterministic policy is significantly worse than the sampled stochastic policy, indicating the probability mass is spread across actions and argmax picks a suboptimal one. **v11 fix:** restore `frame_stack=4` (frame stack handles fast motion, LSTM handles slow patterns — complementary not redundant); reduce `n_epochs=4→2` to control LSTM drift; lower `learning_rate=1e-4→5e-5` for stability; extend `total_timesteps=4M→6M` to compensate for slower LSTM convergence. The "LSTM owns all memory" hypothesis (v10) was worth testing but the evidence says the step budget is too small for frame_stack=1 on this game. v10 will run to completion for full-curve data.

- 2026-05-20 — **v9 smoke ladder → reward normalization, then full 4M launch.** v9's first 500K smoke ([`ppo_v9_smoke.yaml`](configs/ppo_v9_smoke.yaml), LR 2.5e-4 inherited from v8) caught a value-fitting failure: `explained_variance` pinned at exactly 0 for the full 100K observed, `value_loss` diverging 782→1525, while the policy itself improved (eval 34→126). Diagnosis: the deeper IMPALA net + larger return targets from `action_repeat=8` overwhelm v8's LR. **Fix 1 — LR 2.5e-4 → 1e-4** ([`ppo_v9_smoke_lr.yaml`](configs/ppo_v9_smoke_lr.yaml), extended to 500K): killed the divergence (EV climbed off zero) but EV then *ceilinged at ~0.4* (vs v8's ~0.7) with `value_loss` still ~500–1500 and a noisy, non-trending eval curve (best 718 was a fluke; avg ~300). Key insight: EV is scale-invariant, so the residual problem is the **return-target scale/variance** from AR=8's chunky per-step rewards, not the LR — lowering LR further would only learn *slower*. **Fix 2 — `normalize_reward: true`** (SB3 VecNormalize, `norm_reward` only, `norm_obs=False`; [`ppo_v9_smoke_vecnorm.yaml`](configs/ppo_v9_smoke_vecnorm.yaml)) was decisive: EV 0.4→**0.9**, `value_loss` ~800→**~0.1**, eval return @500K ~245→**2608** on a smooth monotonic climb. Implementation decisions: (1) **VecMonitor sits *inside* VecNormalize** (`VecNormalize(VecMonitor(SubprocVecEnv(...)))`) so `rollout/ep_rew_mean` logs raw reward (comparable across runs) while PPO trains on normalized reward. (2) **`norm_obs=False`** — images are scaled in the CNN forward, and the bare eval env (which reports *raw* returns) would otherwise see a different obs scale; keeping obs raw means train/eval obs distributions match exactly. (3) **`gamma` is threaded from `cfg.ppo.gamma`** — the return-variance estimate is gamma-dependent; a mismatch silently corrupts the scale. (4) **Resume-safe stats**: VecNormalize running averages are saved as a `.pkl` sidecar next to each `step-<N>.zip`/`best.zip` (atomic write, pruned in lockstep) and reloaded via `VecNormalize.load` on `resume_from`; without this a crash-resume would reset the reward scale to fresh. Eval/inference never need the stats (norm_obs=False, no reward norm at inference). Config hygiene: `ppo_v9.yaml` now bakes in LR 1e-4 + `normalize_reward: true`; `ppo_v9_smoke_lr.yaml` pinned back to `normalize_reward: false` so it stays a faithful record of the LR-only rung. 3 new checkpoint tests (pkl written for step+best, no-pkl-without-vecnorm, pkl pruned with zip); the `_FakeModel` stub gained `get_vec_normalize_env`. Full 4M run launched (PID 36682, ~21–22 h). One pre-existing unrelated test failure remains (`test_load_env_config_from_repo_yaml`: stale `survival_bonus` 0.01 assertion vs env.yaml's v8-retuned 0.03).

- 2026-05-18 — **v5 fire mechanic: `AutoFireWrapper` (tap-fire)** (after v3/v4 zero-kill plateau). v3 ran to 200K with `Discrete(9)` always-fire combos (each combo set `B=1`); eval/mean_return dropped from -29.90 (100K, anomalous long episode) to -35.75 (200K) and the stochastic rollout sat flat at the no-kill survival floor. Diagnosis followed two false leads (passive corner-hiding via `survival_bonus`, then clip-ceiling masking `score_delta`) before user observation of the eval video — "aircraft barely shoots" — flagged a deeper issue. **True root cause**: Airstriker fires only on the *rising edge* of button B; holding B continuously fires exactly one bullet at the press event, then nothing. With every combo's `B=1`, the agent was holding B for the entire episode — one bullet per life. Verified empirically with [`scripts/diagnose_fire_button.py`](scripts/diagnose_fire_button.py): held B → score 0 over 600 frames; tap pattern (1 on, 3 off) → consistent score. This also explains why v2 worked despite the Bernoulli threshold — stochastic sampling at P(B=1)≈0.14 naturally toggled B and produced press-release cycles. Fix: new [`AutoFireWrapper`](src/retro_rl/env/wrappers.py) wraps the raw env (innermost in the stack) and overrides the fire bit on a per-emulator-frame schedule (1 frame on, 3 off, period=4) regardless of what the policy or DiscreteActionWrapper emits. Because it wraps the raw env, it sees every frame inside `ActionRepeat`'s skip loop — tap cadence is independent of action_repeat. Combos updated to encode movement only (B=0 throughout, 9 directions); reward shaping reverted to v2-style (`score_delta=1.0`, `clip=[-50, 10]`, `survival_bonus=0.01`) since the score signal now actually flows. Implementation: ~30 LOC wrapper + `AutoFireConfig` pydantic model + `EnvConfig.auto_fire: AutoFireConfig | None`. Tests: 8 new (7 for `AutoFireWrapper` + composition, 1 for `AutoFireConfig` validation), full suite 31 env tests passing. Integration smoke: random movement actions now score 20 in 600 frames vs 0 in v3/v4. v1–v4 artifacts preserved for ablation comparison.
- 2026-05-18 — **v3 action space: `Discrete(9)` always-fire** (after v2 Bernoulli-threshold deadlock). v2 ran to 1.66M steps with the new reward shaping; stochastic rollout reached 1174 return (≈119 kill-events/ep) but deterministic eval froze at exactly -35.75 for 8 consecutive checkpoints (1.0M→1.6M). The policy head is now categorical; deterministic argmax selects fire+move directly without any probability threshold. Implementation: ~50 LOC wrapper + `EnvConfig.action_combos: list[list[int]] | None` field; same reward shaping as v2 (it was correct), same PPO hyperparams. Tests: 6 new (5 for `DiscreteActionWrapper`, 1 for config validation); full suite 99 passing. v1 + v2 artifacts preserved for ablation comparison. **NB: This decision turned out to be only half-right — the "always-fire" intent was correct, but the implementation (hardcoded `B=1`) didn't account for tap-fire semantics; see the v5 entry above.**

- 2026-05-19 — **v7 AutoFire period 4 → 24** (after v6 hit the same ~275 ceiling for a different reason). v5 and v6 both topped out because AutoFire at `period=4` (15 Hz) saturates Airstriker's player-bullet sprite array within ~0.6s of each life, after which the game silently drops B presses for the remaining 2-5s.
