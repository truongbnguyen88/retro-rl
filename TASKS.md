## Decisions log

- 2026-05-25 — **v12 — temporal-attention extractor: implemented + smoke launched.** After v11 fell short of v9, weighed three next-steps: (A) extend stateless v9 with more compute, (B) LSTM burn-in to patch v11's cold-start, (C) attention-based memory. Rejected A — v9 already *clears the game*, so more v9 compute has no headroom. Rejected B — burn-in (R2D2, Kapturowski 2019) is an off-policy technique and would only patch an architecture the task doesn't clearly need. Chose C: replace the LSTM with **self-attention over the frame window**, the RL-community direction (attention displacing recurrence) and the one that *eliminates* the cold-start rather than patching it. Resolved a key design fork with the user — three meanings of "attention" in SB3: (1) **temporal-over-frames** (chosen), (2) spatial/ViT over the CNN feature map (different hypothesis), (3) cross-timestep GTrXL memory (high effort, custom rollout buffer). Picked (1): it tests the *same* temporal-memory hypothesis the LSTM did, has no hidden state to initialize, and integrates with **standard PPO** — zero changes to trainer/eval/callbacks/checkpoint/backend (the `algorithm="ppo"` path and `predict()→(action, None)` are reused unchanged). Implementation: `TemporalAttentionExtractor` ([`models/attention.py`](src/retro_rl/models/attention.py)) — shared per-frame Nature-CNN tokenizer (one token/frame) + learned positional embedding + 2-layer/4-head pre-LN TransformerEncoder (full self-attention) + last-token readout → `features_dim`. `d_model = features_dim` (no separate token-dim knob); `n_heads` must divide it. Per-frame encoder is a *lightweight* Nature-CNN (not IMPALA) so K=8 passes ≈ v9's single IMPALA pass in FLOPs. Cold-start eliminated: FrameStack pads all K slots with the first frame at reset ([`wrappers.py:322`](src/retro_rl/env/wrappers.py#L322)) — always-populated window, no `(h,c)=0`. Registered `"temporal_attn"` in `FEATURE_EXTRACTORS`; the config validator + `build_ppo` read the registry automatically (no other wiring). Configs are `extends:` overlays on the v9 best run: `env_v12.yaml` (frame_stack 4→8), `ppo_v12.yaml` (extractor swap only — keeps VecNormalize/lr=1e-4/ent-schedule/4M), `ppo_v12_smoke.yaml` (500K). SB3 detail confirmed: `is_image_space` accepts (84,84,8) uint8 and `VecTransposeImage` transposes to (8,84,84) since H=W=84≫K. Verification: 21 model tests + 104 non-emulator suite green; ruff clean; end-to-end PPO build+predict on a dummy 8-ch env works. Smoke launched (PID 3953, detached). GO gate: EV>0.85, clip_fraction<0.25, eval return ~2608 @ 500K, fps≥~25 (throughput is the K=8 risk; fall back to frame_stack=6 or lr=5e-5 if needed). Risk acknowledged: v9 clears the game with a 533ms window, so the longer attended window may add nothing — the smoke's return-vs-v9 bar surfaces a null result before 4M is committed.

But the eval curve was **noisy and bimodal** the whole way (std_return 800–2160; episodes either ~1700 steps or ~400–600). The three standout evals (5110 @ 5.0M, 4890 @ 5.3M, 5330 @ 5.7M — all `mean_length`≈1650–1760) only appeared in the **final 1M steps**, once `ent_coef→0.001` collapsed entropy and closed the stochastic/deterministic gap. With `(h,c)=0` at every episode reset, frame_stack=4 supplies fast-motion context but the LSTM's slow-timescale memory must re-accumulate each life; "bad" eval episodes are ones where it never recovers (the bimodality). v9's stateless policy has no such per-episode failure mode — every episode starts equally. - 2026-05-23 — **v10 mid-run assessment @ 2.8M steps → plan v11 with frame_stack=4 + LSTM hybrid.** v10 at 2.8M: `eval/mean_return=1993`, `eval/mean_length=555`, `clip_fraction=0.363`, `eval/std_return=1141`. For comparison, v9 at 3M: `eval/mean_return=7028`, `eval/mean_length=4500`. Three failure modes identified. **(1) frame_stack=1 cold-start blindness:** with `h=0,c=0` at episode reset, the LSTM has zero temporal context for the first several decision steps — the agent flies blind on every new episode. A single 84×84 grayscale frame contains no velocity information; the LSTM must infer motion direction from sequential observations, but this takes multiple steps to accumulate. v9 had immediate velocity context via frame stacking. This is the primary driver of `mean_length=555` (dying fast). **(2) Clip_fraction=0.363 oscillation:** stuck elevated since 1.2M steps (0.35 then, 0.36 now — no improvement). RecurrentPPO's LSTM re-execution drift accumulates across the 4 optimization epochs each rollout, causing the importance ratio `r_t = π_θ/π_θ_old` to drift outside [0.9,1.1] on a large fraction of transitions. The policy commits to a strategy, overshoots, gets clipped, oscillates. This is a structural problem with `n_epochs=4` for recurrent policies. **(3) Stochastic/deterministic gap:** rollout return 3311 vs eval return 1993 (1318-point gap).
